# AGENTS.md

## Scope

These instructions apply to the entire repository.

## Purpose

Letterlink is a Vite + React + TypeScript SPA for turning uploaded font glyphs into configurable lettering boards that can be exported as SVG files for cutting or printing. The app is fully client-side and persists working state in `localStorage`.

Use this file as the primary repository contract for coding agents. Keep `README.md` human-friendly, and treat `CLAUDE.md` as a compatibility pointer rather than a second source of truth.

## Product flow

1. Load the default project on first visit or import a font / saved project.
2. Prepare the glyph set by adjusting anchor points and accent placement.
3. Configure names, board dimensions, and render settings.
4. Preview paginated boards and export single-board or ZIP SVG output.

## Architecture

- `src/App.tsx` is the app shell. It wires localization, project state, glyph editing state, configurator state, and workspace switching.
- `src/hooks/useLetterlinkProject.ts` owns project import, autosave, default-project bootstrap, human-readable feedback, and saved project download behavior.
- `src/hooks/useGlyphEditorState.ts` owns selected glyph state, anchor placement, accent dragging, and glyph editor interactions.
- `src/hooks/useConfiguratorState.ts` owns input parsing, CSV handling, board settings, render settings, pagination, and export status messaging.
- `src/features/project-workflow/` contains the prepare workflow UI.
- `src/features/configurator/` contains the board configuration and preview UI.
- `src/components/` is for small shared UI pieces used across workspaces.
- `src/lib/` is for reusable non-UI logic such as import/export, rendering, glyph geometry, persistence helpers, and i18n.
- `src/types/` contains shared type exports used across features and utility modules.

## Current source-of-truth behavior

- Default project bootstrap comes from `public/default-project.json`, generated by `npm run generate:default-project`.
- Autosaved project state is stored under `letterlink-active-project-v2`.
- Saved language preference is stored under `letterlink-language-v1`.
- Localization strings and typed app feedback live in `src/lib/i18n.ts`.
- SVG document generation and project-file import/export live in `src/lib/export.ts` and `src/lib/svg.ts`.
- Glyph extraction and geometry processing live in `src/lib/glyphs.ts` and `src/lib/glyph-geometry.ts`.
- Board rendering and pagination flow through `src/lib/render.ts` and `src/lib/layout.ts`.

## Engineering rules

- Keep React components presentational when possible. Business rules and data transforms belong in hooks or `src/lib/`.
- Do not introduce React or DOM dependencies into `src/lib/` modules unless the file already exists for browser-specific rendering work.
- Prefer extending existing modules before creating new top-level folders or parallel architecture.
- Keep state ownership in the three existing hooks unless there is a clear architectural reason to add a new top-level state layer.
- Preserve strict TypeScript typing. Avoid `any` unless there is no viable alternative.
- Use `import type` for type-only imports where possible.
- Reuse the existing i18n and typed feedback patterns instead of hardcoding new UI strings in components.
- For UI-facing changes, keep layout and visual styles in `src/styles/` or the existing global stylesheet rather than inline styles.
- When adding new docs or assistant guidance, update the consolidated source instead of duplicating architecture notes across files.

## Change placement

- New workflow UI: extend the relevant feature folder in `src/features/`.
- New shared visual element: add it to `src/components/`.
- New reusable stateful behavior: extend one of the existing hooks first.
- New pure helper or algorithm: add or extend a module in `src/lib/`.
- New shared type: add it in `src/types/`.
- New default values or presets: update `src/lib/constants.ts`.
- Build or deployment changes: update `package.json`, `vite.config.ts`, `wrangler.toml`, `.github/workflows/`, or `scripts/` as appropriate.

## Validation

Before hand-off or commit, run:

1. `npm run lint`
2. `npm run build`

For UI-facing changes, include a screenshot in the PR or hand-off when tooling is available.

## Common pitfalls

- Do not document or reference deleted files or stale concepts such as `src/lib/glyph-config.ts`, `ProjectStatusStrip`, or project `readiness`.
- Do not mutate the active project directly inside components; route project edits through the existing hook APIs.
- Do not skip `npm run build`; type errors can surface there even when the editor looks clean.
- Do not assume a backend, API, or server-side persistence exists. This is a client-only app.
